# Managing Technical Debt

Technical debt is inevitable. The key is to manage it, not eliminate it.

## What is Technical Debt

Like financial debt, technical debt is a trade-off:
- You go faster now by cutting corners
- You pay interest later (slower development, bugs, fires)

The goal is not zero debt. The goal is the right amount of debt.

## Types of Technical Debt

### 1. Prudent and Deliberate

You know the shortcut, you know the cost, you accept it.
We ship this now and refactor after we validate the market.

### 2. Prudent and Inadvertent

You made the best decision with the information you had.
Now you know better. Refactor with the new knowledge.

### 3. Reckless and Deliberate

You know its wrong, you do it anyway, and you dont plan to fix it.
This is the most dangerous kind. Dont do this.

### 4. Reckless and Inadvertent

You dont know what you are doing. You ship bad code unknowingly.
Fix this through code review, mentorship, and learning.

## How to Track Debt

Maintain a technical debt register:

| Item | Impact | Effort | Interest | When to Fix |
|------|--------|--------|----------|-------------|
| No tests on payment flow | Critical | 2 weeks | 3 bugs/month | Next sprint |
| Deprecated library X | Medium | 2 days | Security risk | This sprint |
| Monolithic DB | High | 3 months | Scaling limit | Q3 planning |

## When to Pay Debt

You should pay debt when:

1. It is slowing down feature development
2. It is causing bugs in production
3. It is a security risk
4. The cost of fixing is less than the cost of keeping it
5. You are working in this area anyway (boy scout rule)

## When NOT to Pay Debt

Dont fix debt when:

1. The code is being replaced soon
2. Fixing introduces more risk than value
3. The team has higher priority work
4. It is working fine (performance, reliability, security)

## Strategies

### 1. Boy Scout Rule

Always leave the code cleaner than you found it.
Not your mess? Clean it anyway. Five minutes at a time.

### 2. Dedicated Debt Sprints

Every third sprint, the team fixes technical debt.
No new features, only cleanup.

### 3. Refactoring During Features

When adding a feature in a messy area, refactor first.
Write tests, clean up, then add the feature.

### 4. Automatic Detection

Use tools to track debt automatically:
- Code coverage metrics
- Cyclomatic complexity
- Linting rules
- Dependency freshness
- Build time

## Debt as a Tool

Technical debt is not always bad. Deliberate, temporary debt
lets you ship fast and validate learning. The key is:
- Know what debt you are taking
- Plan when to pay it
- Track the interest
