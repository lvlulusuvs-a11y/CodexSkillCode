# Getting Things Done as an Engineer

Shipping is the ultimate metric. Here is how to ship consistently.

## The 80/20 Rule

80% of the value comes from 20% of the work.
Identify the 20%. Do that first.

For a feature, the 20% is:
1. Working core logic
2. Basic error handling
3. One happy path test
4. Documentation for the API

The remaining 80% is:
1. Edge cases
2. Performance optimization
3. Comprehensive tests
4. Advanced documentation
5. UI polish

Ship the 20%. Iterate on the 80%.

## Breaking Down Work

Big tasks are paralyzing. Break them down.

Bad: "Implement payment system"
Good:
1. Define payment data model
2. Integrate with Stripe API
3. Implement charge flow
4. Implement refund flow
5. Add error handling
6. Add tests
7. Add monitoring

Each subtask is < 4 hours of work.
Do one subtask per session.

## Dealing with Interruptions

1. Log the interruption request
2. Finish your current task (5 min rule)
3. Context switch
4. Log back in to original task

The 5 minute rule: if a task takes less than 5 minutes,
do it immediately. Otherwise, log it and come back.

## Writing Design Docs

Before coding a complex feature, write a one-page design doc:

1. Problem statement (what and why)
2. Proposed solution
3. Alternatives considered
4. Trade-offs
5. Implementation plan

Show the doc to a peer before writing code.
Feedback on a doc costs 15 minutes.
Feedback on code costs days.

## Estimation

Rules of thumb:

1. Small task: 1-3 days (estimate +0%)
2. Medium task: 1-2 weeks (estimate +50%)
3. Large task: 1-3 months (estimate +100%)
4. Huge task: 3+ months (dont estimate, break it down)

## Shipping Cadence

Ship small changes frequently:
- PRs under 400 lines
- 1-3 PRs per day
- Each PR is one logical change
- Deploy daily

This reduces risk and increases feedback.
