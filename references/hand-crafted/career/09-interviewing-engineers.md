# Interviewing Engineers: What I Look For

I have interviewed hundreds of engineers. Here is what matters.

## What I Look For

1. Can they code? -> coding exercise
2. Can they design? -> system design
3. Can they communicate? -> technical discussion
4. Can they learn? -> past projects and growth
5. Can they work with others? -> behavioral questions

## The Coding Interview

Format: pair programming, real IDE, 45 minutes

Bad: Algorithm puzzles (reverse a binary tree on a whiteboard)
Good: Real coding with a familiar language and tools

Example exercise:
"Add a feature to this existing codebase:
Implement a rate limiter that allows 100 requests per minute per user."

What I evaluate:
- Code structure and readability
- Error handling
- Edge cases
- Testing
- Communication during coding

## The System Design Interview

Format: whiteboard or diagram, 45 minutes

Example:
"Design a URL shortening service like bit.ly"

What I evaluate:
- Requirements gathering
- Data model design
- API design
- Trade-off awareness
- Scalability thinking
- Communication

## Behavioral Questions

1. "Tell me about a project you shipped that you are proud of"
2. "Describe a time something went wrong in production"
3. "How do you handle disagreeing with a teammate?"
4. "What is the hardest technical problem you solved?"

What I evaluate:
- Honesty (nobody has perfect projects)
- Learning from mistakes
- Collaboration skills
- Technical depth

## Red Flags

1. Cannot explain their own projects deeply
2. Blames others for failures
3. Never made a mistake (dishonest)
4. Cannot take feedback
5. Talks badly about previous employers
