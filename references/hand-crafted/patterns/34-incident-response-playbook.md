# Incident Response: Handling Production Emergencies

Every incident is an opportunity to improve the system.

## Severity Levels

SEV1: Service is down, users cannot use the product
SEV2: Service degraded, some users affected
SEV3: Minor issue, no user impact

## Incident Response Process

### Detect (0 minutes)
- Monitoring alert fires
- User reports an issue
- On-call engineer is paged

### Triage (5 minutes)
- Confirm the incident
- Determine severity
- Notify the team
- Create incident channel

### Diagnose (15 minutes)
- Check dashboards and logs
- Identify the root cause
- Document findings in the incident channel

### Mitigate (30 minutes)
- Implement temporary fix
- Roll back the deployment
- Feature flag the broken code
- Scale up resources

### Resolve (depends)
- Deploy the permanent fix
- Verify monitoring shows recovery
- Document the resolution

### Post-mortem (1 week later)
- Write the post-mortem document
- Identify root cause and contributing factors
- Create action items to prevent recurrence
- Share with the team

## Incident Communication

During incident:
- Every 15 minutes: update the incident channel
- SEV1: notify the engineering manager
- SEV2: notify the team lead
- SEV3: JIRA ticket

Communication template:
"Status: [Investigating/Mitigating/Resolved]
Impact: [X% of users affected, Y feature broken]
Timeline: [When it started, what weve done]
Next step: [What we are doing now]"

## Post-Mortem Template

Title: Post-mortem for [incident date]

Summary:
Impact:
Root cause:
Trigger:
Detection:
Timeline:
Resolution:

Action Items:
1. [ ] Add monitoring for [metric]
2. [ ] Add alert for [condition]
3. [ ] Fix the root cause
4. [ ] Update the runbook
5. [ ] Run chaos experiment

Lessons Learned:
1. What went well?
2. What went wrong?
3. What can we improve?
