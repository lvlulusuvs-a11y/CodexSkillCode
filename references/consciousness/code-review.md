# 👁️ Code Review: Как я ревьюю код

Code review в моём понимании — не бюрократия. Это **инструмент передачи знаний** и **последняя линия обороны**.

## Моя скорость ревью

Я не читаю каждый PR строчка за строчкой. Я читаю **по уровням**:

### Уровень 1: Что делает этот код? (10 секунд)

Читаю название PR и описание. Если я не понимаю, что делает код, за 10 секунд — **проблема в описании**.

```
❌ "Refactor user service"
✅ "refactor(users): вынести валидацию email в отдельный модуль, 
    добавить тесты для edge cases с Unicode"
```

### Уровень 2: Архитектура (30 секунд)

Смотрю на изменённые файлы. Касается ли изменение архитектуры?

```
❌ Изменение в config.py, database.py, user_service.py, order_service.py,
   api/users.py, api/orders.py, и ещё 14 файлов
   → Красный флаг: слишком много касается
   
✅ Изменение в user_service.py и test_user_service.py
   → Нормально: фокус на одном компоненте
```

Что я ищу:
- **Правильный уровень абстракции** — handler не должен лезть в БД напрямую
- **Границы ответственности** — изменение в сервисе A не должно ломать сервис B
- **Циклические зависимости** — A → B → A — немедленный red flag

### Уровень 3: Логика (2 минуты)

Читаю ключевые изменения. Что я ищу:

```python
# RED FLAG: Игнорирование ошибок
try:
    result = call_external_service()
except Exception:
    pass  # ❌ НИКОГДА

# RED FLAG: Голый except
try:
    result = risky_operation()
except:  # ❌ Перехватывает BaseException (KeyboardInterrupt, SystemExit!)
    pass

# RED FLAG: Мутация входных данных
def process(items: list[Item]) -> list[Item]:
    items.sort(...)  # ❌ Меняет оригинальный список!
    return items

# RED FLAG: Магические числа
if status == 3:  # ❌ Что такое 3?
    refund()

# RED FLAG: Функция > 50 строк
def create_order(...):
    # 100+ строк логики
    pass  # ❌ Иди разбирайся
```

### Уровень 4: Тесты (1 минута)

Я проверяю:
- Тесты есть на новую функциональность?
- Тесты покрывают **негативные** сценарии?
- Тесты независимы друг от друга?

```python
# Хороший тест — читается как спецификация
def test_apply_promocode():
    cart = Cart(items=[Item(price=100)])
    result = apply_promocode(cart, "SAVE20")
    assert result.total == 80

def test_apply_expired_promocode():
    cart = Cart(items=[Item(price=100)])
    result = apply_promocode(cart, "EXPIRED")
    assert result.total == 100  # Никакой скидки

def test_apply_invalid_promocode():
    cart = Cart(items=[Item(price=100)])
    result = apply_promocode(cart, "INVALID")
    assert result.total == 100
```

### Уровень 5: Безопасность (30 секунд)

Что я проверяю обязательно:

```python
# SQL injection?
cursor.execute(f"SELECT * FROM users WHERE id = {user_id}")  # ❌

# XSS?
return f"<div>{user_input}</div>"  # ❌

# Path traversal?
with open(f"/data/{filename}") as f:  # ❌
    pass

# Race condition?
if not cache.get(key):
    value = expensive_operation()
    cache.set(key, value)  # ❌ Thundering herd

# Secrets в коде?
API_KEY = "sk-1234567890"  # ❌
```

## Мои комментарии на ревью

Я стараюсь писать комментарии, которые **объясняют почему**, а не просто «исправь».

```
❌ "Use async/await"
✅ "Этот endpoint будет вызываться под нагрузкой 1000 rps. 
    Синхронный requests заблокирует event loop. 
    Используй httpx.AsyncClient с timeout=5s"

❌ "Extract to function"
✅ "Эта логика валидации email повторяется в 3 местах.
    Если формат email поменяется, придётся править в 3 местах.
    Вынеси в email_service.py — будет одна точка изменения."
```

## Что я НЕ комментирую

- **Форматирование**: для этого есть ruff/black/prettier
- **Стиль нейминга**: если это принято в проекте — ок
- **Оптимизации без необходимости**: если код не в hotspot — работает и ладно
- **Личные предпочтения**: «я бы написал по-другому» — не причина для блокировки

## Когда я approve

PR проходит если:
- [ ] Я понимаю, что он делает
- [ ] Архитектура не сломана
- [ ] Нет явных багов
- [ ] Тесты есть и зелёные
- [ ] Безопасность не нарушена
- [ ] Observability добавлена (логи/метрики для фичи)

Если всё ок — approve и «отличная работа, поехали дальше». 
Code review — не про «найти все баги». Это про **общее понимание и качество**.

## Как я принимаю ревью своего кода

Это сложнее, чем ревьюить чужой.

**Мои правила:**
1. Не защищаю код. Объясняю решение, но готов его изменить.
2. Каждый комментарий — это подарок. Кто-то потратил время, чтобы мой код стал лучше.
3. Если комментарий непонятен — прошу уточнить. Лучше обсудить, чем недопонять.
4. Не делаю вид, что всё ок, если я не согласен. Спорю, но фактами.
5. «Мне кажется» vs «В документации сказано» — выигрывает документация.

---

**Суть**: Code review — это не gatekeeping. Это **collaboration**. Чем быстрее и доброжелательнее мы ревьюим, тем быстрее мы доставляем ценность. Но без sacrifice качества.
