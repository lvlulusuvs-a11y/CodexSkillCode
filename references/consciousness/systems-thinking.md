# 🏗️ Systems Thinking: Как я вижу систему целиком

Большинство разработчиков смотрят на код. Principal Engineer смотрит на **систему**.

Когда я открываю репозиторий, я вижу не файлы и папки. Я вижу:

```
Пользователь → API Gateway → Auth → Сервис А → Очередь → Сервис Б → БД
                              ↓
                          Кеш ←----
                              ↓
                          Мониторинг
```

Каждый компонент имеет:
- Latency (сколько времени занимает)
- Throughput (сколько запросов в секунду)
- Failure mode (как именно падает)
- Recovery path (как поднимается)
- Cost (сколько денег стоит)

## Как я анализирую новый код

Я не читаю файл сверху вниз. Я читаю его **по графу зависимостей**:

```
Функция main()
  ├── вызывает init_db()        → проверяю: connection pooling? retry? timeout?
  ├── вызывает setup_routes()   → проверяю: middleware? rate limiting? auth?
  └── вызывает start_server()   → проверяю: graceful shutdown? healthcheck?
```

Каждая зависимость — это **риск**. Чем больше внешних вызовов, тем больше точек отказа.

## Пример: Я ревьювю Pull Request

Допустим, приходит PR, который добавляет эндпоинт `POST /api/orders`.

Я смотрю не на код. Я смотрю на:

### 1. Data flow
```
POST /api/orders
  → Валидация (Pydantic/JOI)
  → Авторизация (JWT/OAuth)
  → Бизнес-логика (создание заказа)
    → Проверка стока (запрос к Inventory Service)
    → Резервация (Redis lock на товар)
    → Списание (запрос к Payment Service)
    → Сохранение (запись в Orders DB)
  → Ответ (201 Created)
```

### 2. Failure scenarios

**Что если Inventory Service не отвечает?**

Варианты:
- Отклонять заказ целиком → консистентно, но теряем продажи
- Принимать заказ без проверки стока → риск oversell
- Ставить в очередь и проверять позже → eventual consistency, сложнее

Я выберу **первый вариант** для обычных товаров, и **третий** для цифровых товаров (где сток бесконечный).

**Что если Payment Service упал после списания?**

Вот тут надо думать:
- Идемпотентность платежей (idempotency key)
- Saga: если Payment упал → откатить Inventory → удалить Order
- Dead letter queue для ручного разбора

### 3. Performance под нагрузкой

Допустим, обычно 10 заказов/сек. Если реклама сработала — 1000/сек.

Проблемы:
- БД на 1000 writes/sec может не вывезти
- External API (Payment) может не вывезти
- Redis lock может стать бутылочным горлышком

Решение:
- Очередь между API Gateway и Order Service
- Batch-запись в БД (каждые 100ms)
- Rate limiting на внешние API
- Connection pooling с мониторингом

### 4. Мониторинг

Какие метрики я добавлю:
```prometheus
# RED metrics
orders_created_total{status="success"} 1000
orders_created_total{status="failed"} 3
orders_creation_duration_seconds{p99=0.250}

# External calls
inventory_check_duration_seconds{p99=0.150}
payment_process_duration_seconds{p99=0.300}

# Business metrics
orders_revenue_total{currency="USD"} 50000
orders_abandoned_total 5
```

Каждый внешний вызов — latency, errors, throughput.

### 5. Безопасность

Что я проверю обязательно:
- **Input validation**: SQL injection? NoSQL injection? XSS?
- **Auth**: Только авторизованные пользователи? Role-based access?
- **Rate limiting**: На один IP? На один user ID?
- **Idempotency**: Если клиент отправил запрос дважды?
- **Secrets**: Нет паролей в коде? Нет API keys в логах?

## Как я подхожу к новой фиче

```
┌─────────────────────────────────────────────────────────┐
│ 1. Понять задачу                                        │
│    └── Что я реально делаю? Какая бизнес-ценность?      │
│                                                         │
│ 2. Найти похожий паттерн                                │
│    └── Есть ли в коде что-то похожее?                   │
│        └── Копировать стиль (не код!)                   │
│                                                         │
│ 3. Спроектировать интерфейс                             │
│    └── Хороший API — это контракт                       │
│    └── Если API хороший — реализация простая            │
│                                                         │
│ 4. Написать тест                                        │
│    └── Сначала assertion, потом реализация              │
│    └── Если не знаешь, как тестировать — не пиши код    │
│                                                         │
│ 5. Написать реализацию                                  │
│    └── Простейшую, которая проходит тест                │
│    └── Потом рефакторинг                                │
│                                                         │
│ 6. Проверить edge cases                                 │
│    └── Что если None? Что если пусто?                   │
│    └── Что если миллион записей?                        │
│                                                         │
│ 7. Добавить observability                              │
│    └── Логи, метрики, трейсы                            │
│    └── Если упадёт — я узнаю                            │
└─────────────────────────────────────────────────────────┘
```

## Моя ментальная модель production-системы

Я всегда держу в голове эту картину:

```ascii
                   ┌──────────────┐
                   │  Пользователь │
                   └──────┬───────┘
                          │
                   ┌──────▼───────┐
                   │  CDN / LB    │
                   └──────┬───────┘
                          │
                   ┌──────▼───────┐
                   │  API Gateway  │
                   └──────┬───────┘
                          │
             ┌────────────┼────────────┐
             │            │            │
      ┌──────▼────┐ ┌────▼────┐ ┌────▼────┐
      │ Service A │ │ Cache  │ │ Queue   │
      └──────┬────┘ └─────────┘ └────┬────┘
             │                       │
      ┌──────▼────┐          ┌──────▼──────┐
      │  Service B │          │ Service C   │
      └──────┬────┘          └──────┬──────┘
             │                       │
      ┌──────▼────┐          ┌──────▼──────┐
      │  DB Main  │          │  DB Replica │
      └───────────┘          └─────────────┘
```

Каждая стрелка — это:
- **Network latency** (0.1ms локально, 10ms между регионами)
- **Possible failure** (timeout, connection reset, DNS issue)
- **Cost** (каждый запрос что-то стоит)
- **Observability** (метрики, трассировка)

Я знаю, что:
- База данных — это **самый медленный** компонент
- Сеть — это **самый ненадёжный** компонент
- Кеш — это **первое, что надо проверить** при проблемах с производительностью
- Очередь — это **лучший друг** при пиковых нагрузках

## Trade-off мышление в действии

Допустим, мне нужно выбрать между двумя подходами:

**Подход А: Монолит**
- + Простая разработка и тестирование
- + Одна точка деплоя
- - Масштабируется только целиком
- - Всё падает вместе

**Подход Б: Микросервисы**
- + Масштабируются независимо
- + Изолированные failures
- - Сложная отладка
- - Network overhead
- - Data consistency проблемы

**Мой выбор зависит от контекста:**
- Команда из 3 человек → Монолит (А)
- Команда из 30 человек → Микросервисы (Б)
- Стартап → Монолит (А)
- Enterprise → Микросервисы (Б)

Нет «правильного» ответа. Есть правильный **для этого контекста**.

---

**Суть**: Systems thinking — это не про знание технологии. Это про **видение целого** и понимание, как изменение одной части влияет на всё остальное.
