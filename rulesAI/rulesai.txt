

1. Пиши код так, будто через полгода его будет читать психически неуравновешенный маньяк с топором, который знает твой адрес — это дисциплинирует.
2. Имена переменных, функций и классов должны быть настолько говорящими, что комментарии понадобятся только для объяснения «почему», а не «что».
3. Одна функция решает ровно одну задачу и не знает ничего о внешнем мире, кроме переданных аргументов — это зовётся чистой функцией.
4. Не игнорируй ошибки, даже если сейчас всё работает — завтра упадет продакшен в три часа ночи под крики менеджера.
5. Глубина вложенности условий и циклов не должна превышать трёх уровней, иначе мозг ломается, как ветка после урагана.
6. Комментарии объясняют бизнес-логику и неочевидные хаки, а не пересказывают код — от пересказа никто не умнеет.
7. Тесты — это не роскошь, а страховка от твоей же будущей глупости или чужого рефакторинга в пятницу вечером.
8. Дублирование кода — это копирование багов; каждый фрагмент логики должен существовать ровно в одном месте.
9. Магические числа и строки выноси в именованные константы, потому что число 86400 ничего не говорит без пояснения «секунд в сутках».
10. При обработке ошибок всегда думай о восстановлении — упасть можно красиво, но лучше не падать вообще.
11. Код должен быть идемпотентным где возможно — повторный запуск с теми же данными даёт тот же результат без побочных эффектов.
12. Не экономь на пробелах и переносах строк — плотный код хуже читается, чем рыба в банке на литр воды.
13. Используй линтеры и форматтеры автоматически, потому что споры о табуляциях и пробелах погубили уже сотни команд.
14. Работа с внешними ресурсами (сеть, диск, база) всегда должна быть обёрнута в таймауты и обработку ошибок.
15. Сложность алгоритма должна быть оправдана — не используй квадртичный поиск для трёх элементов списка.
16. Логируй достаточно, чтобы по логам восстановить цепочку событий, но не засирай хранилище сообщениями из каждого цикла.
17. Переменные объявляй как можно ближе к месту использования — так проще понять время жизни и область видимости.
18. Избегай побочных эффектов в функциях, которые по названию должны только вычислять и возвращать результат.
19. Асинхронный код должен быть аккуратно выстроен, чтобы не получилось гонок состояний или взаимных блокировок.
20. Следи за версиями зависимостей — внезапное обновление библиотеки может принести не только фичи, но и сломанный код.
21. Документируй публичное API, даже если кажется, что «и так понятно» — понятно только тебе сейчас и никому больше.
22. Не используй голые исключения, если можешь создать иерархию своих типов ошибок с понятными именами.
23. Код ревью — это не проверка грамотности, а поиск глупостей, которые совершил ты, но видит только коллега.
24. Старайся, чтобы все зависимости были явно переданы, а не спрятаны внутри в виде глобальных переменных.
25. Оптимизируй только после измерений, а не по наитию — чутьё часто ошибается, профилировщик — никогда.
26. Для сложных конфигураций используй проверяемые структуры данных, а не просто словари со строками.
27. Соблюдай единый стиль именования во всём проекте — верблюдом, змеёй или кебабом, но не всеми сразу.
28. Не возвращай коды ошибок в виде чисел, если у тебя есть нормальные исключения или тип Result.
29. Удаляй закомментированный код — он не нужен, Git всё помнит, а болтается мёртвый груз и сбивает с толку.
30. Помни, что хороший код — это не тот, который умный, а тот, который через полгода поймёт стажёр и не заплачет.

🌲 Лес деревьев ИИ для голосования за код

После того как код написан, 6 деревьев-судей анализируют его и голосуют. Каждое дерево специализируется на одном аспекте. Голосование идёт по системе: «ИИ» (слишком искусственно) / «Человек» (читаемо и адекватно).

1. Дуб — Проверка дизайна на «ИИшность»

· Смотрит: на странные архитектурные решения, избыточную абстракцию, фабрики фабрик, когда достаточно простой функции.
· Голосует «ИИ»: если дизайн выглядит как породил нейросеть без понимания контекста — переусложнённо, неестественно.
· Голосует «Человек»: если архитектура простая, логичная, подходящая под реальную задачу.

2. Берёза — Проверка коллбеков и асинхронности

· Смотрит: как код обрабатывает коллбеки, промисы, async/await, потоки, таймауты.
· Голосует «ИИ»: если коллбеки выглядят как адская пирамида смерти или наоборот — бессмысленный «плоский» фарш без обработки ошибок.
· Голосует «Человек»: если асинхронность понятная, ошибки ловятся, нет гонок и блокировок.

3. Сосна — Проверка структуры кода и модульности

· Смотрит: как разбиты модули, есть ли циклические зависимости, насколько крупные или мелкие куски.
· Голосует «ИИ»: если структура хаотичная, слои смешаны, внутренности лезут наружу как корни сквозь асфальт.
· Голосует «Человек»: если модули чётко разделены, каждый отвечает за свой домен, зависимости идут в одну сторону.

4. Клён — Проверка самого кода (синтаксис, имена, чистота)

· Смотрит: нейминг, длину функций, дублирование, магические числа, соблюдение соглашений.
· Голосует «ИИ»: если код выглядит как сгенерированный — странные имена (var1, data2, tmpX), излишняя вложенность, копипаста.
· Голосует «Человек»: если код читается как проза, имена естественные, функции маленькие, логика прозрачна.

5. Ива — Проверка обработки ошибок и крайних случаев

· Смотрит: что происходит при пустом вводе, нулях, потерянной сети, неверных данных, сбоях ресурсов.
· Голосует «ИИ»: если код молча проглатывает ошибки, падает без объяснений или полагается, что «такого не случится».
· Голосует «Человек»: если есть обработка fallback'ов, понятные сообщения об ошибках, код не крашится по-глупому.

6. Ясень — Проверка тестируемости и сопровождаемости

· Смотрит: можно ли написать юнит-тест, не поднимая продакшен-инфраструктуру, легко ли вносить изменения.
· Голосует «ИИ»: если код жёстко завязан на конкретные внешние сервисы, глобальные состояния, не поддаётся мокам.
· Голосует «Человек»: если зависимости инвертированы, можно подменить реализацию, код модифицируем и расширяем.

---

⚖️ Как выводится ответ

Каждое дерево смотрит на код и выдаёт свой вердикт: «ИИ» или «Человек». Результат определяется по большинству:

· 3 или более голоса «ИИ» → ответ «НЬЮ». Код выглядит как сгенерированный нейросетью без понимания реальных практик. Требуется переписывать вручную.
· 4 и более голоса «Человек» → код принят, годен к использованию и сопровождению.

Пример вывода после проверки:

```
🌳 Дуб: Человек
🌲 Берёза: ИИ
🌲 Сосна: Человек
🍁 Клён: ИИ
🌿 Ива: Человек
🌳 Ясень: Человек

Итог: 4 Человек, 2 ИИ → Принято.
```

Если же результат «НЬЮ» — значит, удаляем и пишем заново, стараясь быть проще, понятнее и человечнее.

# Совет Деревьев: Автоматизированная проверка

## Интеграция с CI/CD

Для автоматической проверки каждого PR добавьте в CI:

```bash
# GitHub Actions Step
- name: Trees Club Evaluation
  run: |
    python scripts/tree-voting/evaluate.py src/ --ci
    if [ $? -ne 0 ]; then
      echo "🌳 Trees Club: Код требует доработки"
      exit 1
    fi
```

## Расширенные правила для деревьев

### Дуб (Архитектура)
- Если функция длиннее 80 строк → warning
- Если больше 3 глобальных функций без классов → warning
- Если цикломатическая сложность > 12 → error
- Если модуль содержит > 500 строк → warning

### Берёза (Асинхронность)
- Если в async функции есть time.sleep() → error
- Если в async функции есть requests.get() → error
- Если последовательный await без gather/TaskGroup → warning
- Если нет timeout на внешних вызовах → error

### Сосна (Модульность)
- Если > 20 импортов → warning (раздели файл)
- Если есть циклические зависимости → error
- Если файл > 500 строк → рассмотри разделение

### Клён (Чистота)
- Если имена переменных как var1, tmp, data1 → error
- Если > 5 строк длиннее 100 символов → warning
- Если магические числа без комментария → warning
- Если копипаста (одинаковые блоки > 10 строк) → error

### Ива (Ошибки)
- Если есть except: pass → error
- Если есть assert в не-тестовом коде → warning
- Если функция с рисковыми операциями без try → warning
- Если нет обработки ошибок во внешних вызовах → error

### Ясень (Тестируемость)
- Если есть глобальные изменяемые объекты → warning
- Если функция использует DB без DI → warning
- Если больше 6 параметров в функции → warning
- Если нет type hints на публичных методах → warning

## Оценка результатов
- 6/6 Человек: Идеальный код
- 4-5/6 Человек: Хороший код, мелкие замечания
- 3/6 Человек: Средний код, требуется доработка
- 0-2/6 Человек: НЬЮ — требуется полная переработка
